# Sharjah Council — Unified Security Oversight Dashboard

**Customer:** Sharjah Cybersecurity Council (مجلس الشارقة للأمن السيبراني)
**Scope:** Pillar 1 of the executive briefing — custom posture & maturity dashboard built on Microsoft Graph across 100+ Sharjah government entity tenants.
**Partner:** Microsoft Security (co-engineered)
**Stack:** Microsoft Graph · Azure · Power BI / Web · Sentinel (for Pillar 2 ingestion)

## What this is

A federated command layer above 100+ entity tenants. The Council measures, ranks, and directs; entities operate. This repo holds the design, architecture, and eventually the code for **Pillar 1 — the Posture & Maturity Dashboard**.

See the executive briefing: [`~/Desktop/Sharjah-Council-Executive-Briefing-final.pptx`](../Sharjah-Council-Executive-Briefing-final.pptx) — slide 6 defines this build.

  **runHuntingQuery — `Query` (capital Q) → `query` (lowercase).** The single biggest reason Mizan reported "Defender Vulnerability Management not available" + "advanced hunting failed" on every E5-licensed tenant: `lib/graph/signals.ts` was sending the request body as `{ Query: "..." }` (capital Q). - **2026-04-27 — v2.5.17 (SQLite moves off NFS — fixes the v2.5.16 perf cliff)**. v2.5.16 stopped corrupting the database, but the price was crippling: every SQLite read on NFS goes through `fcntl` byte-range locks (NLM round trips, ~10–20ms each over Azure Files NFS) plus `synchronous = FULL` doubled commit latency. Page loads that stack hundreds of small queries per request stretched into multi-second waits. The right structural fix isn't to keep tuning pragmas — it's to get SQLite off the network filesystem entirely.

  **Architecture in v2.5.17:**
  - **Live SQLite file** lives on the container's local `EmptyDir` volume at `/local-data/scsc.sqlite`. Full POSIX semantics, microsecond locks, WAL mode safe again. ~50–100× faster than v2.5.16 on Azure.
  - **NFS Azure Files mount** at `/data` is now a **backup target only**, plus the home for write-once assets (uploaded logos, branding files). It no longer hosts any SQLite operation that the request path waits on.
  - **Backup loop** (in `lib/db/client.ts`) calls `db.backup()` every 5 minutes via better-sqlite3's transactional online-backup API. Atomic write via tmp + rename so a partial backup never replaces a good one. Configurable interval via `MIZAN_DB_BACKUP_INTERVAL_MS`.
  - **Boot-time restore**: `EmptyDir` is wiped on every revision swap, but `restoreFromBackupIfNeeded()` runs before SQLite opens — copies the latest `/data/scsc.sqlite` snapshot into `/local-data/` so the new pod boots with the previous pod's last-saved state.
  - **Graceful-shutdown final backup**: SIGTERM handler takes one final `db.backup()` before exiting. ACA's 30-second SIGTERM grace period is plenty. Combined with the 5-minute periodic loop, soft restarts (deploys, scale events, revision swaps) lose zero data; only a SIGKILL hard-crash can lose up to N minutes of writes.

  **Bicep changes** (idempotent — safe to re-run on existing resource groups): added a second volume of type `EmptyDir` mounted at `/local-data`. Container env vars now split: `DATA_DIR=/data` (NFS, persistent) for logos and the backup directory, plus `SCSC_DB_PATH=/local-data/scsc.sqlite` and `MIZAN_DB_BACKUP_DIR=/data` for the new architecture.

  **Existing v2.5.6 → v2.5.16 ACA deployments**: re-run the v2.5.17 Bicep template on your existing resource group. The next revision will boot, find no SQLite at `/local-data/scsc.sqlite`, restore from `/data/scsc.sqlite` (your existing live DB, which becomes the backup), and serve from local disk. **Zero data loss on the migration.** Verify the upgrade by checking container logs for `[mizan/db] restored from backup: …` on first boot, then `[mizan/db] backup loop started: every 300s …`.

  **Defensive fallback**: the `isOnNetworkedFs()` detector from v2.5.16 is kept. If an operator misconfigures `SCSC_DB_PATH` back to the NFS mount, SQLite opens with the slow-but-safe DELETE-mode pragmas and logs a warning — preventing corruption even on a misconfigured deployment.

  **Mac installer + self-hosted Docker** are unaffected — they don't set `MIZAN_DB_BACKUP_DIR`, so the backup loop short-circuits. SQLite already lives on durable local disk in those topologies; no second copy needed.

Operators couldn't see the actual HTTP status, the Graph error code, or the response body without digging through `audit_log` JSON. - **2026-04-27 — v2.5.16 (SQLite-on-NFS hardening — eliminates the WAL-mode corruption mode that surfaced after two consecutive self-upgrades)**. **The bug that destroyed a real test deployment.** After tag-and-redeploy of v2.5.14 then v2.5.15 in quick succession, `mizandirective3` started 500'ing on every page. The container log showed the textbook SQLite-on-networked-FS death spiral: `SQLITE_IOERR_SHORT_READ` → `SQLITE_NOTADB` → SIGTERM. Root cause was three independent settings that combined to be unsafe; v2.5.16 fixes all three.

  **What actually happened.** ACA's revision-update model is rolling: when a new revision is applied (e.g. via the self-upgrade button), the new pod boots BEFORE the old one terminates, traffic flips when the new pod's readiness probe passes, then the old pod drains. There's a ~30–90 second window where TWO pods have the SQLite database file open on the same NFS-mounted share. That overlap is the hot zone. WAL journal mode (which `lib/db/client.ts` set unconditionally since the project's first commit) uses `mmap`'d shared memory (`*.sqlite-shm`) that the SQLite project itself documents as **incompatible with networked filesystems** — the two pods' shared-memory views diverge and writes to one tear pages the other is reading. Worsened by `maxReplicas: 2` in the Bicep, which let ACA's auto-scaler spin up a SECOND concurrent pod under traffic load, also writing to the same DB. Six days of luck running on this configuration ran out the moment the user shipped two upgrades close enough together that a write hit during the overlap window.

  **Three coordinated fixes:**

  1. **`lib/db/client.ts` — auto-detect networked FS, switch to safe journal mode.** New `isOnNetworkedFs()` reads `/proc/mounts`, finds the longest mount-point prefix containing the DB directory, returns `true` if the fs type is `nfs/nfs4/cifs/smb/smb3/smbfs`. Mac/Windows hosts (no `/proc/mounts`) short-circuit to `false`. ACA's `nfsAzureFile` mount is caught (mounts as `nfs4`). When networked FS is detected: `journal_mode = DELETE` (per-write rollback journal, no shared memory — safe over NFS), `synchronous = FULL` (one extra fsync per commit, full durability), `busy_timeout = 30000` (30s wait for lock instead of erroring with SQLITE_BUSY). Local disk: keeps WAL with `synchronous = NORMAL` and `busy_timeout = 5000` (faster, safe). Operators on exotic setups can force the safe mode via `MIZAN_DB_NETWORK_FS=true`.

  2. **`web/deploy/azure-container-apps.bicep` — pin to one replica.** Was `maxReplicas: 2` (allowed concurrent pods writing to SQLite — fundamentally incompatible). Now `minReplicas: 1, maxReplicas: 1`. SQLite is single-writer by design; horizontal scale-out belongs to a future Postgres migration (see backlog), not duct-taped onto the existing schema. Mizan's write workload (a few writes per sync cycle, single-customer-per-deployment) doesn't need >1 replica.

  3. **`web/deploy/azure-container-apps.bicep` — explicit single-revision mode.** Set `configuration.activeRevisionsMode: 'Single'`. Was relying on the api-version's default — which IS `Single` for `2024-10-02-preview`, but being explicit guards against future api-version bumps reverting to `Multiple`. In Multiple mode, ACA keeps the old revision serving alongside the new one for traffic-shifting; in Single mode it's killed when the new one becomes active. Single mode means a brief blip (~5 seconds) instead of overlap, which is a price worth paying to never have two SQLite writers.

  **JSON template regenerated** via `az bicep build` — the CI guard from v2.5.13 keeps these in sync from now on.

  **For existing v2.5.6 → v2.5.15 ACA deployments**: re-run the v2.5.16 Bicep template against your existing resource group. It's idempotent — it patches the running container app's `maxReplicas` and `activeRevisionsMode` in place, no recreate needed. Then upgrade to v2.5.16 image (which auto-detects NFS and switches journal mode at next DB open). Order matters: do the Bicep patch FIRST so the new image isn't running on the still-bad multi-replica config during boot. **If your DB is already corrupted** (`SQLITE_NOTADB` in logs), the file is unrecoverable in place — restore from backup if you have one, or wipe `/data/scsc.sqlite` + rerun `/setup`. The Entra apps Mizan auto-provisioned still exist in your tenant; you'll need to either add a fresh client secret to them in Entra portal or let the wizard create new apps.

  **Postgres migration was considered and deferred.** Initial reaction was to migrate the database engine entirely, treating SQLite-on-ACA as a fundamentally lost cause. After git history showed NFS has been the storage choice since BEFORE v2.5.6 — meaning the configuration ran clean for 6 days — it became clear the corruption was triggered by specific contributory bugs (WAL pragma + multi-replica scale + revision overlap), not by SQLite being inherently incompatible with the deployment. A 2-day refactor wasn't warranted to solve a fixable misconfiguration. Postgres remains the right destination if/when Mizan needs horizontal scale-out, multi-operator concurrent writes, or a multi-tenant SaaS pivot — full architecture spec kept in the project backlog memory file for resumption.

Root cause: `web/Dockerfile`'s runtime stage copied `.next/`, `node_modules/`, `public/`, `package.json`, and `lib/db/schema.sql` — but NOT `assets/`. `lib/pdf/fonts.ts` reads `assets/fonts/Inter-Regular.ttf` and `assets/fonts/NotoKufiArabic-Regular.ttf` via `process.cwd()` at request time (the `@react-pdf/renderer` `Font.register({ src })` API requires real disk paths, so Next can't bundle them). On every Azure deployment from v2.5.0 → v2.5.6 the missing files made `ensureFontsRegistered()` throw `Missing PDF font: /app/assets/fonts/Inter-Regular.ttf`, which Next surfaced as an opaque 500. Fix: (1) Dockerfile's runtime stage now copies `/app/assets ./assets`; (2) `lib/pdf/fonts.ts` tries multiple resolution bases (`process.cwd()`, walks up from `__dirname`, `/app`, `/home/site/wwwroot`, plus a new `MIZAN_ASSETS_DIR` env override) and throws a single line listing every path it tried — so future "missing font" errors are debuggable from a log line, not a guess; (3) the three PDF route handlers (`/api/discovery-letter`, `/api/tenants/{id}/onboarding-letter`, `/api/docs/{id}`) now catch render exceptions and return JSON `{error:"pdf_render_failed", message:"..."}` with the full actionable message instead of letting Next render an opaque 500 page. Browser tab navigation onto a PDF endpoint that fails now shows the JSON error body, which carries the actual reason (missing font, unrenderable AR string, etc.).
