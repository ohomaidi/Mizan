# Sharjah Council — Unified Security Oversight Dashboard

**Customer:** Sharjah Cybersecurity Council (مجلس الشارقة للأمن السيبراني)
**Scope:** Pillar 1 of the executive briefing — custom posture & maturity dashboard built on Microsoft Graph across 100+ Sharjah government entity tenants.
**Partner:** Microsoft Security (co-engineered)
**Stack:** Microsoft Graph · Azure · Power BI / Web · Sentinel (for Pillar 2 ingestion)

## What this is

A federated command layer above 100+ entity tenants. The Council measures, ranks, and directs; entities operate. This repo holds the design, architecture, and eventually the code for **Pillar 1 — the Posture & Maturity Dashboard**.

See the executive briefing: [`~/Desktop/Sharjah-Council-Executive-Briefing-final.pptx`](../Sharjah-Council-Executive-Briefing-final.pptx) — slide 6 defines this build.

## Status

- **2026-04-27 — v2.5.8 (Mac/Windows installer downloads, PDF signature box layout, logo background-strip removed)**. Three changes that arrived together.

  **One-click upgrade for Mac/Windows native installs.** Previously the About panel only had a real upgrade UX on Azure Container Apps — Mac and Windows installs were shown a misleading `docker pull` snippet (those installs aren't Docker; they're native binaries managed by LaunchAgent / Windows Service). v2.5.8 wires up a "Download Mizan-X.Y.Z.pkg" / ".msi" button that points at the GitHub Release asset for the operator's platform. The release workflow (`.github/workflows/release.yml`) now runs three jobs per tag: `build-and-push` (Docker image + GitHub Release, unchanged), `build-mac-pkg` (on `macos-latest`, runs `deploy/mac-build.sh`, attaches `mizan-X.Y.Z.pkg` to the release), and `build-windows-msi` (on `windows-latest`, runs `deploy/windows-build.ps1` with WiX, attaches `mizan-X.Y.Z.msi`). Both installers are upgrade-safe in place: Mac postinstall does `launchctl bootout` + `bootstrap` so the running LaunchAgent picks up the new code automatically; Windows uses WiX MajorUpgrade + `<ServiceControl Stop="both"|Start="install">` so the service stop → file replace → start sequence is automatic. `DATA_DIR` is outside the install root in both cases (`~/Library/Application Support/posture-dashboard/` on Mac, `%ProgramData%\Mizan\data\` on Windows) so SQLite + config + uploaded logo survive every upgrade. Both installers also now copy `assets/fonts/` into the install root (same Dockerfile bug we fixed in v2.5.7 — PDFs would 500 on Mac/Windows installs without the fonts). `/api/updates` runtime detection extended from `aca|docker|unknown` to `aca|mac|windows|docker|unknown` via a `MIZAN_RUNTIME` env var the LaunchAgent / Service wrapper sets; falls back to `process.platform` for older installs that predate the marker. New `installerUrl` field on the response carries the direct GitHub Release asset URL for the operator's platform.

  **Logo background-strip removed.** The U-2-Net + onnxruntime-node auto-removal feature flaked across deployments — the 4.6 MB ONNX model wasn't always shipped (depending on installer + Dockerfile state), the native binding fired "Failed to load dynamic library" on cold-start in some Azure deployments, and the mask routinely chipped real logo content (thin strokes, gradient edges) so operators were re-uploading anyway. Removed: `lib/branding/remove-bg.ts`, `public/models/u2netp.onnx` (4.6 MB saved per image), `onnxruntime-node` from package.json (large native binding gone), the "keep background" toggle in `BrandingPanel`, the `keepBackground` form field in the upload route. New `lib/branding/logo-image.ts` exports just `normalizeLogo` — round-trip through sharp with `ensureAlpha()` + `compressionLevel: 9`. Operators upload pre-cropped PNGs with their own transparency, which is what most were doing anyway. The `logoBgRemoved` field on the branding config is set to `false` always now and ignored on read; kept in the schema for backward compat with existing DBs.

  **PDF signature block kept on one page.** The Onboarding Letter's "5. Sign-off" section was splitting across two pages — heading + name labels on page 2, date fields + signature lines orphaned on page 3. Wrapped the whole sign-off block in `<View wrap={false}>` so `@react-pdf/renderer` reflows it to the next page atomically when it doesn't fit on the current one. Discovery Letter has no equivalent block (no fix needed there).

- **2026-04-27 — v2.5.7 (PDF fonts missing from production Docker image)**. Real-deployment regression: every PDF endpoint (discovery letter, per-tenant onboarding letter, all 5 handoff docs × 2 langs = 13 endpoints) returned "internal server error" on Azure Container Apps deployments. Local Mac demos worked fine because they ran from the source tree. Root cause: `web/Dockerfile`'s runtime stage copied `.next/`, `node_modules/`, `public/`, `package.json`, and `lib/db/schema.sql` — but NOT `assets/`. `lib/pdf/fonts.ts` reads `assets/fonts/Inter-Regular.ttf` and `assets/fonts/NotoKufiArabic-Regular.ttf` via `process.cwd()` at request time (the `@react-pdf/renderer` `Font.register({ src })` API requires real disk paths, so Next can't bundle them). On every Azure deployment from v2.5.0 → v2.5.6 the missing files made `ensureFontsRegistered()` throw `Missing PDF font: /app/assets/fonts/Inter-Regular.ttf`, which Next surfaced as an opaque 500. Fix: (1) Dockerfile's runtime stage now copies `/app/assets ./assets`; (2) `lib/pdf/fonts.ts` tries multiple resolution bases (`process.cwd()`, walks up from `__dirname`, `/app`, `/home/site/wwwroot`, plus a new `MIZAN_ASSETS_DIR` env override) and throws a single line listing every path it tried — so future "missing font" errors are debuggable from a log line, not a guess; (3) the three PDF route handlers (`/api/discovery-letter`, `/api/tenants/{id}/onboarding-letter`, `/api/docs/{id}`) now catch render exceptions and return JSON `{error:"pdf_render_failed", message:"..."}` with the full actionable message instead of letting Next render an opaque 500 page. Browser tab navigation onto a PDF endpoint that fails now shows the JSON error body, which carries the actual reason (missing font, unrenderable AR string, etc.).

- **2026-04-27 — v2.5.6 (one-click ACA self-upgrade + release workflow creates GitHub Releases + updater error states)**. Three related changes that turn the upgrade UX from "copy this Azure CLI command" into "click Upgrade".

  **One-click self-upgrade (the big change).** New `POST /api/updates/apply` endpoint that swaps the running container app's image tag in-place via ARM, using the container app's system-assigned managed identity. Sequence: admin gate → resolve target version (caller-supplied or latest GitHub Release) → mint ARM token from `IDENTITY_ENDPOINT` → GET current container app spec → rewrite every container whose image starts with `ghcr.io/ohomaidi/mizan:` to the target tag → PATCH back to ARM. ACA spins up the new revision and shifts traffic when its readiness probe passes (~1–2 minutes); no downtime. The endpoint refuses sideways/downgrade attempts and only swaps Mizan-image containers (sidecars stay untouched).

  **About panel UX.** Upgrade UX now branches three ways: (a) ACA + self-upgrade ready → big accent-colored **Upgrade to vX.Y.Z** button that opens a confirm dialog and fires `/api/updates/apply`, with green "upgrade requested" / red "upgrade failed" banners surfacing the result; (b) ACA but managed identity not wired up → amber "One-click upgrade not configured" callout pointing at `docs/15-self-upgrade.md` plus the manual `az containerapp update` snippet; (c) Docker / self-hosted → manual `docker pull` snippet (one-click can't work without exposing the Docker socket). Runtime detection lives in `/api/updates` via a new `runtime: "aca" | "docker" | "unknown"` + `selfUpgradeReady: boolean` response shape.

  **Bicep template (`web/deploy/azure-container-apps.bicep`)** wires up everything new deployments need: `identity.type = SystemAssigned`, role assignment granting `Container Apps Contributor` (`358470bc-b998-42bd-ab17-a7e34c199c0f`) on the resource group, and `MIZAN_AZURE_RESOURCE_ID` env var injected from `resourceId('Microsoft.App/containerApps', '${namePrefix}-app-${uniq}')`. New `containerAppPrincipalId` output. Existing deployments enable self-upgrade by either re-running the template (idempotent) or running three `az containerapp identity assign` / `az role assignment create` / `az containerapp update --set-env-vars` commands documented in `docs/15-self-upgrade.md`.

  **Release workflow + updater fixes** (the original bug that surfaced this work). The `release.yml` workflow only built + pushed the docker image to ghcr — it never created an actual GitHub Release, so `/api/updates` queried `/releases/latest` and got 404. Workflow now extracts the matching CHANGELOG entry and calls `gh release create` / `gh release edit` (idempotent for `workflow_dispatch` re-runs). Backfilled v2.5.2 → v2.5.5 GitHub Releases manually so the updater works against existing tags immediately. The `/api/updates` failure fallback used to return `upToDate: true` on any GitHub fetch failure — misleading operators on rate-limited boxes. Now returns `upToDate: false` with a new `checkError` field carrying the reason; the About panel renders an amber "couldn't check" banner instead of a fake "up to date" pill. Failure responses cache for 5 minutes (was 1 hour) so transient blocks don't lock out real updates. Optional `GITHUB_TOKEN` env var bumps the rate-limit from 60/hour to 5,000/hour. New canonical doc: `docs/15-self-upgrade.md` covers architecture, security model (RG-scoped role grant; admin-only endpoint), enablement on existing deployments (Bicep re-run OR three-command patch), every error code with a recovery path, and explicit "no, we won't add this to Docker" rationale.
